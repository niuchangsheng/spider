## 📝 如何创建设计文档

1. **复制模板**
   ```bash
   cp docs/designs/DESIGN_TEMPLATE.md docs/designs/2026-02-XX-your-feature.md
   ```

2. **填写设计内容**
   - 按照模板填写各部分内容
   - 确保包含架构图、接口定义、流程图等

3. **提交设计评审**
   ```bash
   git checkout -b design/your-feature
   git add docs/designs/2026-02-XX-your-feature.md
   git commit -m "docs: 添加XXX功能设计文档"
   git push origin design/your-feature
   # 创建Pull Request，标记为 [DESIGN REVIEW]
   ```

4. **参加设计评审会议**
   - 邀请架构师、Tech Lead参会
   - 讲解设计方案
   - 记录评审意见

5. **根据反馈修改**
   - 根据评审意见修改设计
   - 重新提交评审

6. **批准后更新ARCHITECTURE.md**
   - 将设计内容同步到主架构文档
   - 开始编码实现

## 🔍 设计评审标准

设计文档必须包含以下内容才能通过评审：

- [x] 变更概述和原因
- [x] 现状分析
- [x] 详细的设计方案（架构图、接口定义、流程图）
- [x] 技术方案和核心算法
- [x] 完整的影响分析（性能、兼容性、安全性）
- [x] 实施计划（任务分解、测试计划、文档计划）
- [x] 风险分析和对策

## 📊 设计文档状态说明

- **草稿 (Draft)**: 正在编写中
- **评审中 (In Review)**: 已提交，等待评审
- **已批准 (Approved)**: 评审通过，可以开始实施
- **已实施 (Implemented)**: 已完成编码和测试
- **已拒绝 (Rejected)**: 评审未通过，不予实施

## 💡 最佳实践

### 1. 设计文档要详细但不冗余
- ✅ 包含必要的架构图和流程图
- ✅ 提供清晰的接口定义和示例代码
- ❌ 不要过度设计，保持简单

### 2. 先考虑架构一致性
- ✅ 确保符合现有架构原则
- ✅ 遵循SOLID原则
- ❌ 不要引入与现有架构冲突的设计

### 3. 充分分析影响
- ✅ 性能、兼容性、安全性都要考虑
- ✅ 列出所有受影响的模块
- ❌ 不要忽视边缘情况

### 4. 提供可行的实施计划
- ✅ 任务分解要具体
- ✅ 预估工作量要合理
- ✅ 测试计划要完善

### 5. 及时更新文档状态
- ✅ 设计批准后更新ARCHITECTURE.md
- ✅ 实施完成后更新实施记录
- ✅ 定期回顾已实施的设计
